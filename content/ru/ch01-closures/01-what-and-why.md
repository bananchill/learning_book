## Что такое замыкание

Когда ты создаёшь функцию внутри другой функции, внутренняя функция получает доступ к переменным внешней — даже после того, как внешняя функция завершила выполнение. Вот эта связка — функция плюс её «память» о переменных из места создания — и называется **замыканием** (closure).

Если сказать формально: замыкание — это комбинация функции и ссылок на окружающее её лексическое окружение (lexical environment). На практике это значит, что функция «помнит» переменные, которые существовали рядом с ней в момент создания, куда бы её потом ни передали.

Замыкание — не какой-то особый синтаксис или ключевое слово. Это **механизм языка**, который работает автоматически. Любая функция в JavaScript, которая обращается к переменной из внешней области видимости, является замыканием. Ты пользуешься замыканиями постоянно — возможно, даже не осознавая этого.

## Самый простой пример

```js
function createGreeter(name) {
  // name — переменная внешней функции
  return function () {
    console.log(`Привет, ${name}!`)
  }
}

const greetAlex = createGreeter('Алекс')
greetAlex() // "Привет, Алекс!"
// createGreeter уже завершилась, но name всё ещё доступна
```

Что здесь происходит? Мы вызываем `createGreeter('Алекс')`, она создаёт переменную `name` со значением `'Алекс'` и возвращает анонимную функцию. Казалось бы, `createGreeter` завершилась — её переменные должны исчезнуть. Но возвращённая функция всё ещё ссылается на `name`, поэтому JavaScript сохраняет это значение в памяти. Когда мы вызываем `greetAlex()`, она находит `name` в своём замыкании и выводит `'Привет, Алекс!'`.

## Аналогия с рюкзаком

Представь, что функция — это человек, а переменные внешнего окружения — это вещи. Когда функция создаётся внутри другой функции, она берёт «рюкзак» со ссылками на внешние переменные. Куда бы функция ни отправилась — рюкзак всегда с ней.

Важный нюанс: в рюкзаке не копии вещей, а **ссылки**. Если кто-то изменит оригинал — функция увидит изменения. Это ключевой момент, к которому мы вернёмся ниже.

## Зачем нужны замыкания

### 1. Приватное состояние

В JavaScript нет встроенной приватности для обычных переменных (до `#private` полей в классах). Замыкание — это способ создать переменную, которая доступна только определённым функциям:

```js
function createCounter() {
  let count = 0 // Эта переменная недоступна снаружи

  return {
    increment() { return ++count },
    getCount() { return count },
  }
}

const counter = createCounter()
counter.increment() // 1
counter.increment() // 2
counter.getCount()  // 2
// counter.count — undefined, переменная приватна!
```

Переменная `count` живёт только внутри замыкания. Нет никакого способа обратиться к ней напрямую — только через методы `increment` и `getCount`. Это настоящая инкапсуляция: внешний код не может случайно (или намеренно) перезаписать `count`.

### 2. Фабрики функций

Замыкание позволяет создавать специализированные функции из общего шаблона. Ты передаёшь параметры один раз, а дальше работаешь с «настроенной» функцией:

```js
function multiply(factor) {
  return (number) => number * factor
}

const double = multiply(2)
const triple = multiply(3)

double(5)  // 10
triple(5)  // 15
```

Каждый вызов `multiply` создаёт новое замыкание со своим значением `factor`. `double` навсегда запомнила `factor = 2`, а `triple` — `factor = 3`. Они полностью независимы друг от друга.

### 3. Коллбэки и обработчики событий

Практически каждый обработчик событий в JavaScript — это замыкание. Функция-обработчик создаётся в одном контексте, а вызывается позже, когда происходит событие:

```js
function setupButton(buttonId, message) {
  const button = document.getElementById(buttonId)
  // Коллбэк — замыкание, запоминает message
  button.addEventListener('click', () => {
    alert(message)
  })
}

setupButton('btn-hello', 'Привет!')
setupButton('btn-bye', 'Пока!')
```

Когда пользователь нажмёт кнопку, обработчик найдёт нужное значение `message` в своём замыкании — хотя функция `setupButton` давно завершилась.

## Ссылка, а не копия

Это **ключевой** момент, который часто становится источником багов. Замыкание хранит **ссылку** на переменную, а не её значение в момент создания. Если переменная изменится — замыкание увидит новое значение:

```js
function createFunctions() {
  let value = 1

  function getValue() { return value }
  function setValue(v) { value = v }

  return { getValue, setValue }
}

const obj = createFunctions()
obj.getValue() // 1
obj.setValue(42)
obj.getValue() // 42 — значение изменилось!
```

Обе функции — `getValue` и `setValue` — замкнулись на **одну и ту же** переменную `value`. Это не две копии — это одна переменная, доступная через два замыкания. Изменение через `setValue` мгновенно видно в `getValue`.

Этот принцип «ссылка, не копия» объясняет, почему классический баг с `var` и циклом работает именно так — мы разберём это подробно в разделе про частые проблемы.

## Каждый вызов — новое замыкание

Каждый раз, когда вызывается внешняя функция, создаётся **новое** окружение с **новыми** переменными. Замыкания от разных вызовов полностью изолированы:

```js
function createCounter() {
  let count = 0
  return () => ++count
}

const a = createCounter()
const b = createCounter()

a() // 1
a() // 2
b() // 1 — свой собственный count!
```

`a` и `b` — две разные функции с двумя разными замыканиями. У каждой свой `count`, они не пересекаются. Первый вызов `createCounter()` создал одно окружение, второй — другое.

## Итого

| Факт | Описание |
|------|----------|
| Что это | Функция + ссылка на внешние переменные из места создания |
| Когда создаётся | При создании любой функции (но заметно, когда переменные используются за пределами исходной области) |
| Что хранит | Ссылку на переменную, не копию значения |
| Зачем | Приватность, фабрики, коллбэки, инкапсуляция |
