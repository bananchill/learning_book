## Классика: цикл с var и setTimeout

Это самый известный баг, связанный с замыканиями. Он встречается на каждом втором собеседовании по JavaScript и в каждом третьем code review:

```js
// ❌ Баг: выведет "5 5 5 5 5"
for (var i = 0; i < 5; i++) {
  setTimeout(() => console.log(i), 100)
}
```

**Почему так?** Вспомним, что `var` создаёт переменную на уровне функции, а не блока. Это значит, что все 5 итераций цикла работают с **одной и той же** переменной `i`. Когда `setTimeout` создаёт колбэк, этот колбэк не копирует текущее значение `i` — он сохраняет **ссылку** на переменную. К моменту, когда таймеры сработают (через 100 мс), цикл уже полностью завершился и `i` равна 5. Все 5 колбэков читают одну и ту же переменную — и все видят 5.

### Решение 1: let

```js
// ✅ Выведет "0 1 2 3 4"
for (let i = 0; i < 5; i++) {
  setTimeout(() => console.log(i), 100)
}
```

`let` создаёт **новую привязку** переменной на каждой итерации цикла. Каждый колбэк замыкается на свою копию `i`. Первый колбэк видит `i = 0`, второй — `i = 1`, и так далее. Это самое простое и идиоматичное решение.

### Решение 2: IIFE

До появления `let` (ES2015) использовали немедленно вызываемое функциональное выражение (IIFE). Идея — создать новое окружение с копией текущего значения:

```js
// ✅ Работает и с var
for (var i = 0; i < 5; i++) {
  ;(function (j) {
    setTimeout(() => console.log(j), 100)
  })(i)
}
```

IIFE вызывается на каждой итерации и создаёт новое лексическое окружение с параметром `j`, который получает текущее значение `i`. Колбэк `setTimeout` замыкается на `j`, а не на `i`. Теперь у каждого таймера своя переменная.

### Решение 3: отдельная функция

Тот же принцип, но без IIFE — просто выносим создание таймера в отдельную функцию:

```js
function createTimer(value) {
  return () => console.log(value)
}

for (var i = 0; i < 5; i++) {
  setTimeout(createTimer(i), 100)
}
```

`createTimer` вызывается с текущим `i`, создаёт новое окружение с параметром `value`, и возвращает замыкание над ним.

## Утечки памяти

Замыкание удерживает ссылку на лексическое окружение внешней функции. Если в этом окружении есть тяжёлые объекты — они останутся в памяти до тех пор, пока живёт замыкание:

```js
// ❌ heavyData не будет собран GC
function setup() {
  const heavyData = new Array(1_000_000).fill('x')

  document.getElementById('btn').addEventListener('click', () => {
    console.log('Кнопка нажата')
    // heavyData не используется в коллбэке, но замыкание
    // удерживает ссылку на всё окружение setup()
  })
}
```

Обработчик события живёт столько же, сколько живёт элемент `btn` на странице. Всё это время массив из миллиона элементов висит в памяти, хотя нигде не используется. V8 оптимизирует такие случаи (хранит только реально используемые переменные), но полагаться на это не стоит — лучше явно ограничивать, что попадает в замыкание.

### Как избежать

Извлеки нужные данные **до** создания замыкания:

```js
// ✅ Вынести нужные данные
function setup() {
  const heavyData = loadData()
  const summary = heavyData.length // берём только то, что нужно

  document.getElementById('btn').addEventListener('click', () => {
    console.log(`Элементов: ${summary}`)
  })
  // heavyData может быть собран GC — замыкание не ссылается на него
}
```

### Не забывай removeEventListener

Если элемент удаляется из DOM, а обработчик не снят — замыкание (и всё, что оно держит) остаётся в памяти:

```js
function attachHandler(element) {
  const handler = () => { /* ... */ }
  element.addEventListener('click', handler)

  // Верни функцию для очистки
  return () => element.removeEventListener('click', handler)
}

const cleanup = attachHandler(button)
// Когда обработчик больше не нужен:
cleanup()
```

Паттерн «функция очистки» особенно важен в SPA-фреймворках, где компоненты постоянно создаются и уничтожаются.

## this и замыкания

`this` — это **не** обычная переменная. Замыкание не захватывает `this` из внешней функции. Значение `this` определяется способом вызова функции, а не местом её создания:

```js
const obj = {
  name: 'Объект',
  getName() {
    return function () {
      return this.name // ❌ this зависит от способа вызова
    }
  },
}

const fn = obj.getName()
fn() // undefined (this === window в браузере, или undefined в strict mode)
```

Здесь `getName()` возвращает обычную (не стрелочную) функцию. Когда мы вызываем `fn()` без объекта перед точкой, `this` внутри неё — это `window` (или `undefined` в strict mode), а не `obj`.

### Решение: стрелочная функция

Стрелочные функции — единственное исключение из правила. Они **не имеют собственного this** и берут его из лексического окружения, как обычную переменную:

```js
const obj = {
  name: 'Объект',
  getName() {
    return () => {
      return this.name // ✅ стрелочная функция берёт this из getName
    }
  },
}

obj.getName()() // "Объект"
```

Стрелочная функция создаётся внутри `getName()`, где `this === obj`. Она замыкается на этот `this` и хранит его, как обычную переменную. Вызывай её как угодно — `this` всегда будет `obj`.

## Stale closure в React

React-компоненты — это функции, которые вызываются на каждый рендер. Каждый вызов создаёт **новое** замыкание с текущими значениями state. Проблема возникает, когда старое замыкание используется там, где ожидается актуальное значение:

```jsx
// ❌ count всегда 0 в замыкании
function Counter() {
  const [count, setCount] = useState(0)

  useEffect(() => {
    const timer = setInterval(() => {
      setCount(count + 1) // замыкание захватило count = 0
    }, 1000)
    return () => clearInterval(timer)
  }, []) // пустой deps → эффект создан один раз при монтировании
}
```

Проблема: `useEffect` с пустым массивом зависимостей запускается один раз — при монтировании. Замыкание `setInterval` захватывает `count = 0` из первого рендера. На каждом тике оно вычисляет `0 + 1 = 1` — счётчик навсегда застрял на 1.

```jsx
// ✅ Решение: функциональный updater
setCount((prev) => prev + 1) // не зависит от замыкания
```

Функциональная форма `setCount` получает **актуальное** значение state как аргумент. Замыканию больше не нужно знать текущий `count` — React сам передаст свежее значение.

## Итого: чеклист проблем

| Проблема | Признак | Решение |
|----------|---------|---------|
| Цикл с var | Все коллбэки видят одно значение | `let`, IIFE, отдельная функция |
| Утечка памяти | Обработчик держит тяжёлые данные | Вынести нужное, `removeEventListener` |
| this в замыкании | `this === undefined` | Стрелочная функция |
| Stale closure | Данные устарели (React, setInterval) | Функциональный updater, deps |
